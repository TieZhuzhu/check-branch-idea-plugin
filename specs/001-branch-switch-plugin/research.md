# 研究记录：IDEA 多项目分支快速切换

## 决策 1：使用 IntelliJ Platform Gradle Plugin 2.x

**结论**：使用 `org.jetbrains.intellij.platform` 2.x 作为插件构建体系。

**理由**：JetBrains 官方文档说明 2.x 是当前用于构建、测试、验证、配置环境和发布 IntelliJ 平台插件的推荐 Gradle 插件体系。参考项目 `easy-pojo2json` 也已经使用 2.x，迁移风险低。

**备选方案**：

- `org.jetbrains.intellij` 1.x：参考项目 `mt-tool` 使用过，但 1.x 不再是新项目首选。
- 手写构建脚本：会增加维护成本，不符合最小实现原则。

## 决策 2：使用 Java 21 和 IDEA 2024.2+

**结论**：源码编译目标使用 Java 21，插件最低运行环境为 IntelliJ IDEA 2024.2+。

**理由**：用户已指定 Java 21。IDEA 2024.2 起默认随 JBR21 提供，运行环境与 Java 21 目标更一致。该项目是新插件，不需要为了旧 IDE 版本保留 Java 17 编译目标。

**备选方案**：

- Java 17 + IDEA 2023.3+：兼容面更广，但与用户明确要求冲突。
- Java 21 + 更旧 IDEA：运行环境不一致，插件验证风险更高。

## 决策 3：使用右侧工具窗口作为唯一首版入口

**结论**：首版只提供右侧工具窗口入口，并通过工具窗口中的列表和按钮完成刷新、选择和切换。

**理由**：用户明确希望右下角侧边栏点击后右侧展开固定停靠窗口。JetBrains 工具窗口文档也建议将支持编码过程的工具和服务放入工具窗口，并可通过 `toolWindow` 扩展点注册到右侧。

**备选方案**：

- 主菜单动作：入口更隐蔽，不利于持续查看多仓库状态。
- 弹窗式工具：不适合长时间展示仓库状态和切换结果。

## 决策 4：只发现当前 IDEA 窗口内的 Git 仓库

**结论**：首版只处理当前 IDEA 窗口中已打开或已附加的 Git 仓库。

**理由**：该范围与规格假设一致，也更符合 IDEA 插件上下文。扫描任意磁盘目录会引入权限、性能、误识别和项目导入边界问题。

**备选方案**：

- 扫描用户指定本地根目录：适合后续增强，但会显著扩大首版范围。
- 同时支持项目仓库和磁盘扫描：复杂度更高，不符合最小实现原则。

## 决策 5：优先搁置变更，降级到 Git 暂存栈

**结论**：切换前未提交变更保护优先使用 IDEA 搁置变更能力；不可用或失败时，在用户同意后使用 Git 暂存栈。

**理由**：用户明确提出 IDEA 原生搁置变更优先，Git 暂存栈作为可接受降级路径。两者都不自动恢复，能降低冲突自动处理风险。

**备选方案**：

- 直接阻止所有有变更仓库切换：最安全但效率价值下降。
- 切换后自动恢复：体验看似顺滑，但冲突处理不可控，容易导致误解和风险。

## 决策 6：结果模型以“部分成功”为默认

**结论**：批量操作允许部分仓库成功、部分失败，不自动回滚已成功仓库。

**理由**：多仓库状态差异很大，自动回滚会引入更多失败路径。逐仓库结果可追踪，更适合工具窗口展示。

**备选方案**：

- 全部成功或全部失败：语义简单，但会让单个仓库阻塞所有仓库。
- 自动回滚：复杂度高，且可能与用户本地变更保护产生冲突。

## 决策 7：发布流水线采用构建、验证、上传、标签发布四段式

**结论**：建立 `ci.yml` 和 `release.yml`，持续集成负责构建测试和插件验证，发布流水线负责标签触发、产物上传和 GitHub 发布。

**理由**：用户要求参考 `deep-read-web/.github`。该结构清晰区分日常验证和正式发布，也适合后续接入 JetBrains Marketplace。

**备选方案**：

- 仅本地构建：缺少可追踪发布产物。
- 只做 GitHub Release 不做插件验证：会增加安装失败和兼容性风险。
