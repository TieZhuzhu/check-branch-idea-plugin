# 功能规格：IDEA 多项目分支快速切换

**功能分支**：`001-branch-switch-plugin`

**创建日期**：2026-06-03

**状态**：草稿

**输入**：用户描述：“参考我自己开发的 IDEA 插件项目 mt-tool、easy-pojo2json 进行技术选择；当前项目准备做一个 IDEA 插件，用于在 IDEA 中快速切换多个微服务项目的分支。用户希望在右侧工具窗口展开固定停靠窗口，支持统一或单个项目切换，切换前处理未提交代码，可优先使用 IDEA 搁置变更，也可降级为 Git 暂存栈，并参考 deep-read-web 的流水线发布流程。”

## 用户场景与测试 *(必填)*

### 用户故事 1 - 扫描工作区项目分支状态（优先级：P1）

开发者打开包含多个微服务项目的 IDEA 工作区后，可以在右侧工具窗口看到各项目的当前分支、主分支识别结果、工作区是否有未提交变更，以及是否可参与批量切换。

**优先级原因**：没有可靠的项目与分支状态，后续批量切换会缺少判断依据，用户也无法信任插件的执行结果。

**独立测试**：打开一个包含多个 Git 仓库的工作区，展开插件工具窗口，确认每个可识别项目都显示当前分支、可切换状态和变更状态。

**验收场景**：

1. **假设** IDEA 工作区包含多个 Git 项目，**当** 用户打开插件工具窗口，**则** 系统展示所有可识别项目及其当前分支。
2. **假设** 某个项目不是 Git 仓库或无法读取分支，**当** 用户查看项目列表，**则** 系统标记该项目不可切换并展示原因。
3. **假设** 某个项目存在未提交变更，**当** 用户查看项目列表，**则** 系统在切换前明确提示该项目需要暂存或跳过。

---

### 用户故事 2 - 批量切换到指定分支（优先级：P1）

开发者输入一个目标分支名后，可以对选中的多个项目执行统一切换。若项目存在目标分支，则切换到目标分支；若不存在目标分支，则在确认该项目存在主分支时切换到主分支。

**优先级原因**：这是插件最核心的效率价值，解决微服务工作区逐个切分支耗时的问题。

**独立测试**：在多个测试仓库中准备目标分支存在、目标分支缺失但主分支存在、目标分支和主分支均不可用的项目，执行批量切换并验证每个项目的最终状态和结果提示。

**验收场景**：

1. **假设** 选中的项目都存在目标分支，**当** 用户执行批量切换，**则** 所有项目切换到目标分支并展示成功结果。
2. **假设** 某个项目不存在目标分支但存在主分支，**当** 用户执行批量切换，**则** 该项目切换到主分支并在结果中标记为主分支回退。
3. **假设** 某个项目既不存在目标分支也无法确认主分支，**当** 用户执行批量切换，**则** 该项目不执行切换并展示可操作的失败原因。

---

### 用户故事 3 - 切换前保护未提交变更（优先级：P1）

开发者在切换前遇到未提交变更时，可以选择先暂存变更再切换。暂存优先使用 IDEA 的搁置变更能力；当搁置变更不可用时，系统允许降级使用 Git 暂存栈，并清楚提示用户切换后需要手动恢复变更和处理冲突。

**优先级原因**：分支切换会直接影响本地工作区，未提交变更保护决定插件是否安全可用。

**独立测试**：准备有未提交变更的测试仓库，分别验证可暂存、暂存失败、用户选择跳过、用户选择取消全部切换时的行为。

**验收场景**：

1. **假设** 项目存在未提交变更且用户选择暂存，**当** 系统执行切换，**则** 变更被暂存，项目完成分支切换，并在结果中提示如何恢复变更。
2. **假设** 搁置变更不可用但 Git 暂存栈可用，**当** 用户同意降级暂存，**则** 系统使用 Git 暂存栈保存变更并继续切换。
3. **假设** 用户不希望暂存某项目的变更，**当** 执行批量切换，**则** 该项目被跳过，其他选中且安全的项目继续执行。

---

### 用户故事 4 - 单个项目精确切换（优先级：P2）

开发者可以在工具窗口中只选择某一个项目并输入目标分支进行切换，用于处理个别服务需要不同分支的场景。

**优先级原因**：微服务联调中并非所有服务都总是切到同一个分支，单项目切换能覆盖更细粒度的日常操作。

**独立测试**：选择单个项目执行切换，确认其他项目状态不变，结果只记录该项目。

**验收场景**：

1. **假设** 用户只选择一个项目，**当** 输入目标分支并执行切换，**则** 只有该项目发生分支变化。
2. **假设** 用户取消选择某些项目，**当** 执行批量切换，**则** 未选择项目不发生任何变化。

---

### 用户故事 5 - 查看切换结果与恢复提示（优先级：P2）

开发者完成切换后，可以看到每个项目的成功、回退、跳过或失败状态，并知道哪些项目存在暂存变更、需要手动恢复或解决冲突。

**优先级原因**：批量操作必须给出清晰反馈，避免用户在多个仓库之间猜测最终状态。

**独立测试**：执行包含成功、回退、跳过、失败和暂存的混合切换，确认结果区域逐项展示状态和下一步提示。

**验收场景**：

1. **假设** 批量切换结束，**当** 用户查看结果，**则** 每个项目都有明确状态和最终分支。
2. **假设** 某个项目切换前执行过暂存，**当** 用户查看结果，**则** 系统提示该项目有待手动恢复的暂存变更。

### 边界场景

- 工作区中同时存在 `main` 与 `master` 时，系统优先使用已配置或当前远端默认主分支；无法确认时按 `main`、`master` 的顺序尝试。
- 目标分支只存在远端分支时，系统允许将该远端分支作为可切换目标，并在结果中标记来源。
- 某个项目正在变基、合并、拣选提交或存在冲突时，系统不应继续切换该项目。
- 暂存成功但切换失败时，系统必须保留暂存记录并提示用户手动恢复。
- 部分项目失败时，已成功项目不自动回滚，结果中必须清楚展示混合状态。
- 用户输入空分支名、包含明显非法字符的分支名或只输入空白时，系统不执行切换并提示修正。
- 用户在切换过程中关闭工具窗口时，当前项目会话内必须保留最近一次切换结果；重新打开工具窗口后可查看最近一次结果摘要；关闭 IDEA 项目后不要求跨会话保留。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须提供 IDEA 右侧工具窗口，并允许用户从侧边栏打开和固定停靠。
- **FR-002**：系统必须发现属于当前 IDEA 工作区或项目组的 Git 仓库。
- **FR-003**：系统必须展示每个已发现仓库的当前分支、可切换状态、未提交变更状态和最近一次操作结果。
- **FR-004**：系统必须允许用户输入目标分支名进行切换。
- **FR-005**：系统必须允许用户在切换前选择全部可用仓库或部分仓库。
- **FR-006**：当目标分支本地存在或可从远端分支解析时，系统必须将选中仓库切换到目标分支。
- **FR-007**：当目标分支不存在时，系统必须回退到可识别的主分支。
- **FR-008**：系统必须优先使用项目配置识别主分支；无配置时使用 `main`、`master` 等常见名称。
- **FR-009**：当目标分支和可识别主分支都不可用时，系统必须跳过该仓库。
- **FR-010**：系统必须在切换仓库前检测未提交变更。
- **FR-011**：当发现未提交变更时，系统必须允许用户选择临时保存变更、跳过该仓库或取消操作。
- **FR-012**：当可用时，系统必须优先使用 IDEA 原生搁置变更能力临时保存变更。
- **FR-013**：当原生搁置变更不可用或失败时，系统必须允许降级使用 Git 暂存栈。
- **FR-014**：系统必须不在切换后自动恢复搁置或暂存栈变更；必须提示用户哪些仓库需要手动恢复。
- **FR-015**：系统必须展示每个仓库的操作结果，包括成功、主分支回退、跳过、失败和待手动恢复。
- **FR-016**：系统必须阻止正在未解决合并、变基、拣选提交或冲突状态中的仓库继续切换。
- **FR-017**：系统必须支持不重启 IDEA 即可刷新仓库和分支状态。
- **FR-018**：系统必须提供发布流水线，用于验证插件包并从标签构建发布产物。

### 关键实体

- **工作区仓库**：当前 IDEA 工作区中检测到的 Git 项目目录，包括显示名称、根路径、当前分支、可用目标分支、主分支候选和操作可用性。
- **分支切换请求**：用户发起的一次操作，包括目标分支、选中仓库、未提交变更处理选择和操作开始时间。
- **临时变更记录**：仓库在切换前临时保存变更的记录，包括保存方式、仓库、时间和面向用户的恢复提示。
- **切换结果**：单个仓库的操作结果，包括最终分支、状态、回退信息、失败原因和需要用户执行的后续动作。
- **发布产物**：发布流程生成并附加到持续集成或发布结果中的插件安装包。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：当所有仓库均可切换时，开发者可在 2 分钟内将 10 个仓库切换到同一目标分支。
- **SC-002**：刷新后，混合工作区中至少 95% 的仓库展示准确的当前分支和变更状态。
- **SC-003**：当目标分支缺失时，用户无需逐个打开仓库即可识别哪些仓库回退到了主分支。
- **SC-004**：在用户验证中，开发者无需阅读外部说明即可完成批量切换和单仓库切换。
- **SC-005**：在已验证的分支切换流程中，未提交用户变更不发生丢失。
- **SC-006**：每次标签发布都能产出可安装插件包，并从持续集成或发布页面获取。

## 假设

- 初始版本面向单个 IDEA 窗口中已打开或已附加的多个 Git 仓库，不主动扫描任意磁盘目录。
- 用户可以接受切换后手动恢复搁置变更或 Git 暂存栈内容，并自行处理冲突。
- 初始版本不做自动拉取、自动创建远端分支或跨仓库依赖编排。
- 主分支识别默认覆盖 `main` 和 `master`，后续可通过配置扩展其他主分支名称。
- 批量操作允许部分成功、部分失败，不为已成功项目做自动回滚。
