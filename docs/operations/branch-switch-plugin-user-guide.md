# 分支切换插件用户操作手册

## 适用范围

本文档面向使用 `easy-multi-project-check-branch` 插件的开发人员，说明插件安装、刷新仓库、批量切换、未提交变更保护和结果排查方式。

当前版本适用于 IntelliJ IDEA 2024.2+，主要服务于一个 IDEA 窗口中同时打开多个 Git 仓库的微服务开发场景。

## 安装插件

从源码构建插件 ZIP：

```powershell
$env:JAVA_HOME="D:\java\jdk\jdk21"
$env:Path="$env:JAVA_HOME\bin;$env:Path"
.\gradlew.bat buildPlugin
```

在 IDEA 中安装：

1. 打开 `Settings | Plugins`。
2. 点击齿轮菜单。
3. 选择 `Install Plugin from Disk...`。
4. 选择 `build/distributions/*.zip`。
5. 重启 IDEA。

开发调试时也可以直接运行：

```powershell
.\gradlew.bat runIde
```

## 打开插件

1. 在 IDEA 中打开目标工作区。
2. 确保工作区中的 Git 仓库已经被 IDEA 识别。
3. 在右侧工具窗口栏打开 `分支切换`。

插件只读取当前 IDEA 项目上下文中已识别的 Git 仓库，不会扫描任意磁盘目录。

## 刷新仓库状态

点击 `刷新状态` 后，插件会异步读取仓库状态，界面会显示进度条并临时禁用刷新和切换按钮，避免 IDEA 页面卡死。

仓库列表字段说明：

- `参与切换`：是否纳入本次批量切换。
- `仓库名称`：仓库展示名称。
- `当前分支`：刷新时读取到的当前分支。
- `状态`：目标分支、阻塞状态或本地变更状态。

常见状态说明：

- `正常`：仓库可参与切换。
- `有变更`：仓库存在已跟踪的未提交变更，切换前需要处理。
- `当前已经处在目标分支，已跳过切换`：当前分支已经等于目标分支，执行时不会重复 checkout。
- `目标分支存在`：本地分支存在。
- `目标分支仅远端存在`：远端引用存在，执行时会优先按远端引用切换。
- `目标分支缺失`：目标分支在当前本地和远端引用中均未找到。
- `正在合并`、`正在变基`、`正在拣选提交`、`存在未解决冲突`：仓库处于阻塞状态，不会继续切换。

## 批量切换分支

1. 在 `目标分支` 输入框中输入分支名，例如 `feature/demo`。
2. 点击 `刷新状态`。
3. 勾选需要参与切换的仓库。刷新后默认全选当前可见仓库。
4. 点击 `执行切换`。

切换规则：

- 如果仓库已经在目标分支，直接跳过。
- 如果远端存在 `origin/目标分支`，优先刷新远端引用并切换到目标分支。
- 如果远端刷新失败或远端不可用，会尝试切换本地同名分支。
- 如果目标分支不存在，会按主分支候选顺序回退，例如 `main`、`master`。
- 批量切换允许部分成功、部分失败，不会因为单个仓库失败自动回滚其他仓库。

## 处理未提交变更

插件默认只把已跟踪的未提交变更纳入保护范围。纯未跟踪文件通常不会触发 `有变更`，以尽量贴近 IDEA 自身切分支体验。

当选中仓库存在未提交变更时，执行切换前会弹出提示：

- `使用 IDEA 搁置变更`：优先调用 IDEA Shelf 搁置该仓库范围内的变更，并在搁置成功后回滚工作区变更，再继续切换分支。
- `跳过该仓库`：当前仓库不切换，其他仓库继续处理。
- `取消本次操作`：终止整批切换。

切换完成后，插件不会自动恢复 Shelf 中的变更。结果卡片出现 `待恢复` 时，请在 IDEA 的 Shelf/搁置变更入口中找到以 `检查分支插件搁置记录` 开头的记录，手动恢复并处理可能出现的冲突。

说明：

- Git stash 能力目前仅保留为代码级显式兜底，不在默认产品入口暴露。
- 如果 checkout 会覆盖未跟踪文件，Git 仍可能拒绝切换；此时结果卡片会显示失败，悬浮卡片可查看完整原因。

## 查看切换结果

切换完成后，底部 `最近一次切换` 区域会展示结果卡片。

卡片状态说明：

- `成功`：已切换到目标分支。
- `回退`：目标分支缺失，已回退到主分支候选。
- `跳过`：仓库已在目标分支，或用户选择跳过存在变更的仓库。
- `失败`：切换失败，需要查看失败原因。
- `待恢复`：分支已切换，但切换前变更已被 IDEA Shelf 保存，需要手动恢复。

如果失败原因过长，鼠标悬浮到结果卡片或卡片内文本即可查看完整原因。

## 常见问题

### 为什么有些仓库显示目标分支缺失？

插件刷新时会检查本地分支和远端引用。请先确认 IDEA 或 Git 中已经获取到最新远端引用。当前插件执行切换时会尝试 `fetch --prune origin`，但刷新状态展示仍依赖当前可读引用。

### 为什么未跟踪的 resources、profiles、rebel.xml 没有阻止切换？

这是当前设计选择。插件默认保护已跟踪变更，纯未跟踪文件不触发保护，尽量与 IDEA 自身切分支体验保持一致。

### 为什么点击使用 IDEA 搁置变更后仍可能失败？

常见原因包括：

- IDEA ChangeList 中没有识别到该仓库范围内可搁置的已跟踪变更。
- 仓库存在冲突、合并、变基等阻塞状态。
- IDEA Shelf API 在当前项目上下文中执行失败。
- 搁置成功后 checkout 被未跟踪文件覆盖风险或 Git 其他错误阻止。

失败时请查看结果卡片 tooltip 中的完整原因。

### 切换后会自动恢复变更吗？

不会。插件只负责在切换前保存变更，并在结果中提醒 `待恢复`。恢复动作需要用户在 IDEA Shelf 中手动执行，避免自动恢复带来冲突覆盖风险。

## 已知限制

- 不自动创建缺失分支。
- 不自动恢复 Shelf 记录。
- 不跨 IDEA 项目会话长期保存结果历史。
- 不扫描当前 IDEA 项目以外的磁盘目录。
- 当前主分支候选由项目偏好服务维护，默认优先 `main`、`master`。
